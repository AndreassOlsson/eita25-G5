# README: Java Client and Server Applications

The Java applications are responsible for establishing the secure network tunnel and enforcing access control. The communication between the client and the server must be encrypted and established using the network standard TLS. We configure the server to demand mutual TLS, meaning it is not enough for the server to just prove its identity to the user; the user must also present a valid certificate back to the server. When a user runs the client application, they are prompted to enter their password to unlock their local KeyStore file. This password is only used locally by the Java client to decrypt the file in memory and is never transmitted over the network. Once the private key is unlocked, the Java Secure Socket Extension library takes over and automatically handles the complex cryptographic handshake with the server.

The server application does not verify users by checking a database of passwords or a list of approved public keys. Instead, the CA certificate should be installed in a truststore on both the server and the client. When the client connects, the server looks at the incoming certificate and simply verifies that it bears the mathematical signature of the internal Root CA located in its truststore. Once the server trusts the certificate, it parses the identity string to extract the user's name, role, and hospital division. The server uses these extracted values to enforce the strict read and write permissions for the medical records. Furthermore, the server uses this exact same extracted identity data to feed the audit log, ensuring the audit log should contain information about who did what and when. The server should store all records and respond to client requests, but the actual record files on the server's hard drive do not need to be stored encrypted.
